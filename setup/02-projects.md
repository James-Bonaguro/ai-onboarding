# 02 — Projects

## Objective

Use Projects to create durable, context-rich workspaces by domain.
Projects prevent repeated setup overhead and keep analytical work coherent over time.

## Why Projects matter

Without project separation, context bleeds across unrelated tasks.
That creates noise, inconsistent standards, and avoidable rework.

With Projects, you can enforce stable defaults for each workflow:

- private credit underwriting,
- portfolio monitoring,
- committee memo drafting,
- market/sector surveillance.

## Recommended project architecture

Create a small number of focused Projects rather than many fragmented ones.

Suggested starter set:

1. **Underwriting Lab** — deal screening, risk mapping, covenant stress points.
2. **Portfolio Surveillance** — borrower health checks, early-warning indicators, watchlist logic.
3. **IC Communication** — memo compression, argument clarity, committee-ready narratives.
4. **Research and Themes** — sector theses, macro-to-credit translation, scenario analysis.

Each Project should have distinct purpose and standards.

## Project-level defaults

For each Project, define:

- objective and scope,
- default output formats,
- tone requirements,
- challenge level (how aggressive the model should be in pushback),
- recurring templates.

Treat these as operating parameters, not preferences.

## Operating practices

- keep active work inside the relevant Project,
- pin core instructions at project start,
- periodically prune stale context,
- version meaningful templates as they improve.

Do not use Projects as storage bins; use them as analytical environments.

## Failure signals

Project setup is weak if you observe:

- repeated re-briefing in every chat,
- inconsistent output style across similar tasks,
- model forgetting key standards,
- cross-task contamination of assumptions.

When these appear, tighten scope and defaults.

## Bottom line

Projects are how you industrialize quality.
They convert one-off prompting into repeatable operating systems.
