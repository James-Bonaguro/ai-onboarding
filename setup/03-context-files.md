# 03 — Context Files

## Objective

Context files are leverage assets: they encode durable knowledge once and reuse it across workflows.
They are the difference between repeatedly explaining your world and operating with persistent analytical memory.

## Two context planes: work and personal performance

For high-performing professionals, durable context should cover both:

- **work systems** (mandate, risk framework, templates),
- **personal performance systems** (training, nutrition, skill practice, execution preferences).

The operating model is the same: explicit standards, structured inputs, measurable outputs.

## Why durable personal context matters

High-quality personal context improves:

- **continuity** — fewer resets across sessions,
- **recommendation quality** — advice matches your constraints and standards,
- **planning quality** — plans reflect real schedule and recovery constraints,
- **tailoring** — outputs align to your preferred decision style,
- **habit stickiness** — routines become easier to maintain when plans are specific and trackable.

## What to include

### Work context files

Start with files that materially improve decision quality:

- investment mandate and risk appetite,
- credit policy and underwriting standards,
- covenant framework and definitions,
- sector watchlists and red-flag indicators,
- memo templates and decision rubrics.

### Personal performance context files

Add high-signal files that improve execution consistency:

- `fitness_context.md`
- `nutrition_principles.md`
- `golf_notes.md`
- `personal_operating_preferences.md`

Prioritize relevance over volume.
A focused file set beats a bloated archive.

## File design standards

Each context file should be:

- **specific** — concrete definitions, thresholds, and frameworks,
- **structured** — clear headings, consistent taxonomy,
- **current** — dated and reviewed periodically,
- **actionable** — usable in live analysis without translation.

Avoid dumping raw notes without curation.

## Example snippets

### `fitness_context.md`

```markdown
## Current training structure
- 4 lifting sessions/week (upper/lower split)
- 2 zone-2 conditioning sessions/week
- Primary constraint: travel 1 week/month

## Current objective
- Increase strength-to-bodyweight ratio while keeping recovery stable

## Non-negotiables
- Session length <= 60 minutes on weekdays
- No high-fatigue plan changes during heavy work weeks
```

### `nutrition_principles.md`

```markdown
## Nutrition defaults
- High-protein baseline at each meal
- Mostly whole-food intake, low added sugar
- Alcohol limited to defined social windows

## Decision rules
- Prefer plans that are repeatable over plans that are theoretically optimal
- Flag recommendations that require high daily complexity
```

### `golf_notes.md`

```markdown
## Skill profile
- Strength: driver distance
- Weakness: approach dispersion from 120-170 yards

## Practice constraints
- 2 range sessions/week max
- 1 on-course session on weekends

## Improvement metric
- Track GIR%, proximity trend, and up-and-down rate monthly
```

## How to use in practice

When initiating a task:

1. attach or reference the relevant context files,
2. instruct the model to prioritize those sources,
3. require explicit flags when responses go beyond provided evidence,
4. request a short “missing information” section.

This preserves analytical hygiene in both work and personal domains.

## Governance cadence

Review high-impact files monthly or after major process changes.
Archive outdated versions rather than silently overwriting assumptions.

If context files drift from real operating standards, model performance degrades quickly.

## Bottom line

Context files are not documentation overhead.
They are infrastructure for consistent, high-quality AI-assisted work and personal performance execution.
